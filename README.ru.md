# Eidetic

**Долговременная память для AI-агентов-кодеров — которая отслеживает, когда её знания устаревают.**

> [Что это?](#что-это) · [Зачем](#зачем-eidetic) · [Установка](#установка) · [Как пользоваться](#как-пользоваться) · [Как работает](#как-это-работает) · [Возможности](#ключевые-возможности) · [Сравнение](#сравнение)

## Что это?

Eidetic даёт AI-агенту-кодеру **долговременную память**, которая хранится в обычных Markdown-файлах и ищется гибридным поиском FTS5 + вектор. Один движок, **два вида памяти**:

- **Личная память (PUSH)** — ваши решения, правила и контекст проектов. **Автоинъецируется** в каждую сессию Claude Code (и доступна по запросу из любого MCP-агента) и **компаундится** — обновляет существующие записи вместо россыпи дубликатов.
- **Topic-базы (PULL)** — внешний корпус (доки API, методология, книга), который вы превращаете в изолированную базу и **подключаете только к тем проектам, где она нужна**.

Чем Eidetic отличается от других инструментов памяти: он **отслеживает устаревание знаний** и понижает их в ранге — чтобы больше памяти не делало агента _хуже_ (вот [зачем](#зачем-eidetic) он нужен). Нативен для Claude Code через хуки без настройки; работает с Codex, Gemini, Cursor, Cline и любым MCP-агентом.

_Идейная родословная: [Zettelkasten Luhmann](https://en.wikipedia.org/wiki/Zettelkasten), [Second Brain Tiago Forte](https://www.buildingasecondbrain.com/) и [**LLM Wiki** Karpathy](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) — их **структура** реализована от начала до конца (типизированные страницы, викиссылки, op-log, компаундящий путь записи), плюс поверх — обнаружение объявленных противоречий и суперсессия; полное семантическое truth-maintenance — в роадмапе v6. Подробнее — [Философия дизайна](#философия-дизайна)._

## Зачем Eidetic?

```
Сессия 1:   "Никогда не мокай базу данных в тестах"
Сессия 2:   *мокает базу данных*
Сессия 50:  *всё ещё мокает базу данных*
```

Это **проблема дня 1** — Claude забывает всё между сессиями. Любой инструмент памяти её решает.

Но есть вторая, сложнее:

```
Сессия 35:  "Используй validate_key() для всех проверок API"
            *validate_key() отлично работает, сохраняется в память*

Сессия 50:  *validate_key() две недели назад переименовали в check_auth()*
            *Claude уверенно подставляет старое правило*
            *Агент становится ХУЖЕ от собственной памяти*
```

**Проблема дня 60** — после 500+ файлов памяти устаревшие знания отравляют агента. Больше памяти = хуже результат.

Eidetic атакует её детекцией дрейфа + truth-maintenance: мёртвые викиссылки, устаревание по возрасту, неподтверждённая эскалация агента и объявленные противоречия / суперсессия (`contradicts:` / `supersedes:`) реально понижают карточку в ранжировании. «Тихое» переименование из примера выше — где ничего не объявлено и ссылки целы — требует **семантического** обнаружения противоречий, это v6; до тех пор пример ловится, когда умирает викиссылка, карточка стареет или новая карточка объявляет `contradicts:`.

## Установка

```bash
git clone https://github.com/LARIkoz/eidetic.git && cd eidetic && bash install.sh
```

Одна команда. `install.sh` спросит (Enter = по умолчанию) про **три модели, на которых держится система** — **эмбеддер** (`multilingual` / `english`), **кросс-язычный перевод** (`off` / `auto` / `apple` / `opusmt`) и **модель извлечения карточек** (`sonnet` / `haiku`). Установка через пайп, в CI или агентом остаётся неинтерактивной через env (`EIDETIC_EMBED_PROFILE`, `EIDETIC_QUERY_TRANSLATE`, `EIDETIC_SIGNAL_MODEL`) или примите умолчания.

**Ядро** (FTS5-поиск, автоинъекция, дрейф, экспорт в Obsidian) не требует **ни одного pip install** и работает сразу. Семантический / кросс-язычный поиск добавляет одну зависимость — `pip install fastembed` (см. [Зависимости](#зависимости)).

**Ставите через агента?** Дайте ему ссылку на репозиторий + **[AGENTS.md](AGENTS.md)** (или готовый промт установки из **[docs/prompts.md](docs/prompts.md)**). Проверка в любой момент: `bash ~/.claude/memory-system/bin/doctor.sh`; откат: `bash ~/.claude/memory-system/bin/rollback.sh`.

## Как пользоваться

У Eidetic **два вида памяти**, используются по-разному.

### Личная память (PUSH) — ваши собственные знания

На Claude Code это **автоматически**: инсталлятор подключает хуки, поэтому накопленные правила / решения / контекст **инъецируются** в начале каждой сессии, а новые сигналы **захватываются** в конце. В основном вы просто работаете. Чтобы обратиться к ней явно:

- **Recall** — `/memory-recall "<запрос>"` (или MCP-инструмент `memory_search` из любого агента).
- **Вернуть ответ в память** — `echo "<ответ>" | python3 ~/.claude/memory-system/bin/remember.py "<заголовок>"` сохраняет хороший синтезированный ответ как типизированную страницу (повторный promote по той же теме дописывает `## Update`, не дублирует).
- **Просмотр** — `eidetic export-vault` превращает память в просматриваемую вики Obsidian.

### Topic-базы (PULL) — внешний корпус, подключаемый к проекту

**Topic-база** — изолированный корпус (доки API, методология, книга), который вы запрашиваете по требованию и подключаете только там, где он нужен — без загрязнения личной памяти. Тот же движок, отдельный индекс, свой git-репозиторий.

```bash
python3 ~/.claude/memory-system/bin/base.py init   acme                  # → ~/eidetic-bases/acme-base/
python3 ~/.claude/memory-system/bin/base.py index  acme                  # построить FTS + e5-вектора
python3 ~/.claude/memory-system/bin/base.py attach acme --scope project  # подключить к проекту через MCP
```

- **Полный гайд + контракт для агента:** **[docs/topic-bases.md](docs/topic-bases.md)** — модель хранения, сценарии ингеста, гочи API-доков.
- Встроенный скилл **`/eidetic-base`** соберёт базу за вас из источника (скрейп → страницы → индекс → проверка → подключение).
- База — **отдельный репозиторий** вне дерева eidetic, поэтому ваши корпуса остаются приватными, хотя сам eidetic публичный.

### Готовые промты для агента

Не запоминайте команды — дайте агенту готовый промт из **[docs/prompts.md](docs/prompts.md)**: _установить Eidetic_ · _собрать topic-базу из сайта / книги / API_ · _подключить базу к проекту_.

## Построено на переиспользуемом движке

Ядро эмбеддинга / векторного поиска Eidetic — это версионируемый публичный шов,
**Engine API v1** (`bin/engine.py`). Память, тематические базы и ваши собственные
потребители строятся на одной двери, а не на приватных внутренностях.

```
   Личная память   Тематические базы   Ваш потребитель   ← продукты
          \_________________ | _________________/
                      bin/engine.py                       ← Engine API v1 (дверь)
                           |
                    embed.py · rerank.py                  ← приватное ядро
```

Чтение деградирует (поиск вернёт `[]` и одну причину, если модели нет), сборка
кричит (`EngineUnavailable`), контракт версионируется (`ENGINE_API 1.1`).
См. [docs/engine.md](docs/engine.md) — быстрый старт, таблица контракта и первый
внешний потребитель (навык поиска по трекеру задач на собственном индексе).

Внешние интеграции используют контракт Core/SDK без путей, зафиксированный в
[ADR 0003](docs/adr/0003-core-sdk-boundary.md); низкоуровневый модуль остаётся
поверхностью Core и обратной совместимости.

## Как это работает

```
                НАЧАЛО СЕССИИ (~350мс в тёплом режиме)
                        |
                Переиндексация (FTS5) + Индекс кода (tree-sitter) + Векторный Embed
                        |   └─ W5 громкое самовосстановление: проваленный embed → предупреждение + лог (embed-last.log),
                        |      никогда не уходит молча в темноту; высокий vector lag помечается
                Сборка контекста (~200 правил + проект + недавнее + диагностика дрейфа)
                        |
                Запись в ~/.claude/rules/ (подхватывается автоматически, без ограничений размера)
                        |
                Агент начинает сессию, ЗНАЯ всё


                     СЕРЕДИНА СЕССИИ
                        |
                ЧТЕНИЕ   /memory-recall "запрос"  или  MCP memory_search
                        |    не-EN запрос? -> перевод на английский (опц.: Apple NMT / Opus-MT) -> двойной запрос
                        |    FTS5 + e5 вектор (принудительно для не-EN) + cross-encoder -> RRF merge (min-rank)
                        |    Калиброванная уверенность (двухсигнальный гейт) + предупреждения о дрейфе
                        |    └─ уверенные попадания логируются -> статистика (топ-карточки · мёртвые карточки на удаление)
                        |

                             └─ search-before-write: повторный promote дописывает ## Update, не дублирует


                     КОНЕЦ СЕССИИ (~5с, асинхронно)
                        |

                        |

                        |
```

### Составное ранжирование

Каждый результат ранжируется по формуле:

```
score = relevance x evidence x source x freshness x status

evidence:   validated = 1.0    observed = 0.7    hypothesis = 0.4
source:     user-explicit = 1.0  agent-extracted = 0.5  system = 0.3  imported = 0.3
freshness:  < 30 days = 1.0    > 30 days = 0.5    (дрейф-находка перекрывает: stale 0.5x, broken link 0.8x)
status:     current = 1.0      resolved/fixed = 0.75   superseded/deprecated = 0.35   archived = 0.25
```

Совпадения по ключевым словам учитывают качество попадания. Проверенная, свежая, актуальная, написанная человеком запись поднимается выше старой, непроверенной, извлечённой агентом или замещённой.

## Ключевые возможности

### Детекция дрейфа (v2.5)

Главное отличие Eidetic. Четыре проверки, троттлинг 24ч, файлы не мутируются:

| Проверка                    | Что обнаруживает                                          | Порог                                  |
| --------------------------- | --------------------------------------------------------- | -------------------------------------- |
| **Дрейф викиссылок**        | `[[validate-key]]` ссылается, но файл переименован/удалён | Немедленно                             |
| **Устаревание по возрасту** | Проектная память не обновлялась 30+ дней                  | 30д проект, 60д статус, 90д справочник |
| **Эскалация уверенности**   | 3+ обновления агентом, 0 подтверждений от человека        | 3 события                              |
| **Объявленное противоречие** | Карточка, которую другая объявляет `contradicts:` (или её собственный `contradicted_by:`) | Немедленно, без grace-гейта |

Находки дрейфа штрафуют ранжирование, **умножаясь в фактор свежести**: сломанная викиссылка = 0.8x, устаревание = 0.5x, эскалация уверенности = 0.3x, объявленное противоречие = 0.4x — устаревшая карточка с находкой всегда ниже просто старой, а находка не может карточку *поднять*. Разные находки на одной карточке **перемножаются** (пол 0.1x): карточка с тремя проблемами ниже карточки с одной. Штраф активируется со **второго** обнаружения (прогоны ≥24ч), кроме объявленных противоречий (0.4x — сразу). Штраф снимается, когда проблема пропадает — включая объявленные отношения: пропагация пересчитывается от текущих деклараций при каждой переиндексации, так что удалённая строка `contradicts:`/`supersedes:` снимает штраф на следующем прогоне. Декларация понижает цель только если декларирующая карточка не старее и не ниже по уровню доверия источника; иначе заявка лишь показывается в диагностике, не трогая ранжирование.

### Умная токен-компрессия (v1.3)

160 правил обратной связи в ~6 800 токенах. Раньше в MEMORY.md помещались ~57. Кластеризация по ключевым словам собирает связанные правила; многоуровневое отображение выводит приоритетные правила целиком, остальные — только по имени.

### Гибридный поиск (v2.0, v5.1)

FTS5 для ключевых слов (~50мс). Векторный поиск (multilingual-e5-large, 1024-dim) как фолбек для семантических запросов. Кросс-языковый изначально — запрос на одном языке находит заметки на другом (e5 покрывает ~100 языков). v5.1 заменил MiniLM-384 на e5-large: кросс-языковый paraphrase recall@3 вырос с **25% до 67%** (измерено на RU→EN). Результаты объединяются через Reciprocal Rank Fusion. Если все кандидаты слабые, система возвращает `No confident results`, а не шум.

**Двухсигнальный гейт уверенности (v5.1).** e5 сжимает оценки: кросс-языковое совпадение (~0.83 cosine) неотличимо от тематического шума (~0.83) по одному косинусу. Векторный хит достигает actionable-уверенности только при лексическом подтверждении (общие якорные слова запроса) — высокий recall без ложной уверенности. Штамп модели/размерности на векторном хранилище защищает от молчаливого дрейфа эмбеддера.

Прогрессивный поиск держит широкие запросы компактными. `--detail <id>` возвращает полное содержимое, когда кандидат выглядит релевантным.

### Поиск по коду (v2.2)

Tree-sitter парсит `.py`, `.js`, `.ts`, `.tsx`, `.sh` — каждая функция и класс доступны для поиска по имени или назначению.

### Компаундинг знаний

Перед созданием записи система ищет существующие на ту же тему. Нашла — обновляет, добавляет историю. Не нашла — заводит новый файл. Сопоставление **лексическое, ещё не семантическое** (перекрытие самых редких ключевых слов + porter-стемминг): повтор той же или умеренно перефразированной формулировки компаундится, сильный парафраз в худшем случае **флагуется** в op-log как `possible duplicate of <card>` — но не дублируется молча. Семантический дедуп через векторы — v6.

### Ответ возвращается в память — Promote (v5.3)

Хороший синтезированный ответ не должен умирать в логе чата. **Promote** сохраняет его как **типизированную страницу** — `echo "<ответ>" | python3 ~/.claude/memory-system/bin/remember.py "<заголовок>"`. Search-before-write: повторный promote по той же теме дописывает датированный раздел `## Update`, а не дублирует, а новая страница получает `## Related` с викиссылками к соседям. Это mid-session компаньон к захвату сигналов в конце сессии; тот же путь записи переиспользует будущий импортёр. Каждая запись попадает в grep-совместимый **op-log** — `grep '^## \[' log.md` — весь таймлайн.

Так Eidetic реализует **структуру** [**LLM Wiki** Karpathy](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) от начала до конца: компаундинг страниц, **контракт обслуживания** ([docs/MEMORY-SCHEMA.md](docs/MEMORY-SCHEMA.md)), типизированные страницы (`synthesis` / `concept` / `entity`) и op-log. Поверх работают автоизвлечение, детекция дрейфа и обработка объявленных противоречий / суперсессии. «Метаболизм» вики — замечать *необъявленные* противоречия и синтезировать сразу по многим страницам — это роадмап v6; эта часть «поддерживает себя, а не гниёт» ещё впереди.

### Автозахват в конце сессии (настраиваемый)

В конце сессии LLM извлекает сигналы `Decision:/Rule:/Worked:/Failed:/Knowledge:` из транскрипта и компаундирует их (`source: agent-extracted`, 0.5x вес). Модель по умолчанию — **Sonnet**; раннер деградирует по цепочке CLI — `claude-batch → claude --print → codex-batch → codex exec` — поэтому **любая установка с CLI `claude` _или_ `codex` захватывает сигналы**, конкретная конфигурация не важна.

Настройка через env: `EIDETIC_SIGNAL_CLAUDE_MODEL` (по умолчанию `sonnet`; `=haiku` для экономии) и `EIDETIC_SIGNAL_CODEX_CLI_MODEL` (иначе берётся модель Codex по умолчанию). Для полнокачественного захвата — **Promote** (выше).

### Экспорт в Obsidian (v4.0)

Eidetic превращает память в **вики Obsidian** — хаб `HOME.md`, папки по типам (правила / проекты / справочники), разрешённые `[[викиссылки]]` и карта содержимого.

> **Хранилище — окно для чтения, не движок.** Поиск, инъекция, компаундинг и детекция дрейфа работают на `.md`-файлах + индексах FTS5 / вектор — _ничто не читает хранилище обратно_. Удалите его — Eidetic работает так же; хранилище регенерируется при следующем экспорте. Оно нужно **вам** — чтобы видеть, что знает агент; агент на него не опирается.

**Где находится:** `~/Documents/eidetic-vault/` по умолчанию (передайте путь для переопределения). Инсталлятор подключает Stop-хук + ночное обновление, поэтому хранилище обновляется автоматически — или пересоберите вручную:

```bash
eidetic export-vault                 # → ~/Documents/eidetic-vault/  (по умолчанию)
eidetic export-vault ~/my-vault/     # произвольное расположение
```

**Откройте в Obsidian:** _Open folder as vault_ и укажите эту директорию (авторегистрируется при первом экспорте). `eidetic doctor` выводит расположение хранилища и количество страниц.

Гейт качества фильтрует файлы памяти до проверенного подмножества; `--polish` переписывает карточки в читабельную прозу через LLM.

## Работает с любым агентом

Eidetic **агент-агностичен на уровне памяти** — единое хранилище `.md` + FTS5 + e5-векторный индекс; различается только _поверхность интеграции_:

- **Claude Code** — без настройки. `install.sh` подключает хуки SessionStart/Stop: память **инъецируется** в начале сессии и **захватывается** в конце. Руками ничего делать не надо.
- **Codex · Gemini · Cursor · Cline — любой MCP-агент** — подключите агента к встроенному **MCP-серверу** (`memory_search`, `memory_search_detail`, `export_vault`, …) для recall по запросу + обратной записи. Захват сигналов в конце сессии идёт через CLI `codex`, а не только `claude`.

Привязки к конкретному агенту нет. Автоинъекция — через хуки Claude Code; остальные агенты обращаются к памяти через MCP.

## Зависимости

Eidetic устроен **по уровням** — ядро работает без дополнительных пакетов; семантический поиск добавляет один pip install.

| Уровень                                                                     | Требуется                                                                                                          | Без этого                                                      |
| --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| **Ядро** — FTS5-поиск, автоинъекция, дрейф, компаундинг, экспорт в Obsidian | `bash` + `python3` + `sqlite3` (предустановлены на macOS/Linux)                                                    | — работает полностью                                           |
| **Семантический / кросс-языковый поиск** (гибридный слой e5)                | `pip install fastembed` + ~2.2 ГБ модель e5-large ONNX (автоскачивается в `~/.cache/fastembed` при первом запуске) | откат к FTS — только поиск по ключевым словам                  |
| **Спасение через cross-encoder rerank**                                     | `fastembed` (подтягивает компактный реранкер, лениво)                                                              | кросс-языковые совпадения без общих слов остаются подавленными |
| **Поиск по коду**                                                           | `pip install tree-sitter tree-sitter-python tree-sitter-javascript tree-sitter-bash`                               | функции/классы кода не индексируются                           |

> ⚠️ `install.sh` устанавливает **только ядро** — pip-install fastembed / tree-sitter не запускается. Свежая установка работает только с FTS, пока вы не выполните `pip install fastembed`. Кэш модели e5 должен быть **персистентным** (`~/.cache/fastembed`); если кэш во временной директории — ОС его вычистит и молча отключит векторный поиск.

**Запустите доктора**, чтобы увидеть, какие уровни активны и чего не хватает — он покажет _причину_, по которой вики/хранилище не создаётся:

```bash
bash ~/.claude/memory-system/bin/doctor.sh
```

Проверяет: зависимости, индекс, файлы памяти на диске, вектора + лаг, расположение кэша модели, хуки, вики/хранилище, **op-log + состояние деплоя promote/compound, распределение card-kind и лог проваленных embed-ов (W5)** — с подсказкой для каждого ⚠️/❌.

Платформа: macOS / Linux (использует `fcntl` файловые блокировки).

### Обновления

Фоновая проверка обновлений каждые 6 часов. Когда доступна новая версия:

```
Eidetic update available (a1b2c3d). Run: bash ~/.claude/memory-system/bin/update.sh
```

Обновления сохраняют базы данных, правила и хуки — заменяются только файлы кода.

## MCP-сервер

**Опциональный.** Claude Code в MCP не нуждается — ему хватает хуков + правил + recall-skill. MCP-сервер нужен _другим_ редакторам без хуковой системы Claude Code (Cursor, Windsurf, Cline, любой MCP-совместимый агент) и открывает то же хранилище памяти как набор инструментов:

```json
{
  "mcpServers": {
    "eidetic": {
      "command": "python3",
      "args": ["~/.claude/memory-system/mcp_server.py"]
    }
  }
}
```

7 инструментов: `memory_search`, `memory_search_detail`, `memory_serendipity`, `memory_health`, `memory_reindex`, `memory_lint`, `export_vault`.

## Философия дизайна

Вдохновлён [Zettelkasten Luhmann](https://en.wikipedia.org/wiki/Zettelkasten), [Second Brain Tiago Forte](https://www.buildingasecondbrain.com/) и идеей Karpathy о вики для LLM — [концепцией AI-вики](https://gist.github.com/karpathy/1dd0294ef9567971c1e4348a90d69285) и паттерном [**LLM Wiki**](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f), структуру которого Eidetic реализует от начала до конца (компаундинг страниц, схема, op-log, типизированные страницы; семантическое truth-maintenance — v6). По Karpathy, _«люди забрасывают вики, потому что обслуживание растёт быстрее ценности; LLM не устают»_ — кривая ценности Eidetic растёт с использованием, а не деградирует.

Принципы:

1. **Файлы — это истина.** Индексы производны. БД умерла — пересборка из markdown за <1 секунду.
2. **Каждая запись находима** без знания, где она лежит. Кросс-проектный поиск по умолчанию.
3. **Критические правила не исчезают.** Система гарантирует доставку всех поведенческих правил агенту.
4. **Система поддерживает себя — или умирает.** AI-агент автоматически выполняет индексацию, связывание, поиск и компаундинг. Человек курирует _истинность_, когда хочет — полное самообслуживание (разрешение противоречий, синтез нескольких страниц, авто-подшивка) — цель v6.

## Безопасность

- **Атомарные записи** — `tempfile` + `os.replace()`, устойчивость к крэшам
- **Бэкап/восстановление** — автобэкап перед переиндексацией, автовосстановление при сбое
- **Сериализация блокировок** — файловая блокировка `fcntl` через `bin/lock_runner.py`
- **Защита от инъекций** — извлечение сигналов с валидацией префиксов блокирует попадание шума из транскрипта в память
- **Защита типов** — promote не дописывает контент агента в проверенную пользователем карточку feedback/profile и не затирает её
- **Громкое самовосстановление (W5)** — проваленный embed при старте сессии попадает в `embed-last.log` + выдаёт однострочное предупреждение (без молчаливых аутейджей вектора); высокий vector lag тоже помечается
- **Грейсфул деградация** — отсутствующий индекс откатывается к `head -200 MEMORY.md`
- **Откат** — одна команда, менее 5 секунд

## Дорожная карта

**Выпущено**

- **v5.6.0** — телеметрия использования: пишет, какие карточки поиск реально достаёт → `usage_stats.py` показывает топ востребованных, **мёртвые карточки** (ни разу не показаны → кандидаты на удаление) и охват %. Append-only, fail-open, только хэш запроса (приватно); секция «Usage» в докторе. Выключить — `EIDETIC_USAGE_LOG=off`.
- **v5.5.0** — кросс-языковой перевод запроса (по желанию): неанглийский запрос автоматически переводится и ищется в двух вариантах (родной + переведённый, объединение по лучшему рангу) — recall@3 растёт с **5/8 до 7/8**. Сменные fail-open бэкенды: Apple Translation NMT (на устройстве, macOS 26) · Opus-MT / CTranslate2 (переносимый, офлайн) · codex CLI. По умолчанию выключено.
- **v5.4.0** — доктор честно проверяет выравнивание векторов (убирает враньё gross-lag, которое прятало 99.94% рассинхрон) · профили модели по языку (e5-large / bge-small-en) · доктор показывает, какая модель что делает · `recall_lab` — замер кросс-языкового recall · русский README
- **v5.3.1** — `imported` — уровень доверия low-trust (`0.3`, подготовка к Wave-1 импортёру) · `EIDETIC_SIGNAL_SKIP_CLAUDE` — маршрут извлечения, безопасный при kickout
- **v5.3** — **promote** (ответ возвращается в память как типизированная страница — LLM Wiki Karpathy) · grep-совместимый **op-log** · типизированный `card_kind` (synthesis/concept/entity) · **громкое самовосстановление embed** (без молчаливых аутейджей) · [схема памяти](docs/MEMORY-SCHEMA.md) · `doctor` покрывает всё
- **v5.2** — спасение через cross-encoder rerank · персистентный кэш модели · блокировки конкурентности embed/export · экспорт хранилища с поддержкой fenced-code · самопроверка `doctor`
- **v5.1** — эмбеддер e5-large + двухсигнальный гейт точности + защита от дрейфа модели (кросс-языковый recall@3 25% → 67%)
- **v5.0** — прогрессивный поиск (+ v5.0.1 lifecycle Phase B)
- **v4.0–4.3** — экспорт в Obsidian + LLM-полировка · сигналы жизненного цикла
- **v2.x** — гибридный поиск (v2.0) · поиск по коду через tree-sitter (v2.2) · детекция дрейфа (v2.5)
- **v1.x** — FTS5 + сигналы + компаундинг (v1.0) · токен-компрессия 2.17x (v1.3)

**Далее** — дистрибуция: pip-пакет, доработка документации.

**В планах — v6 (truth-maintenance)** — *объявленные* противоречия (`contradicts:`/`contradicted_by:`) и суперсессия (`supersedes:`) уже всплывают как находки дрейфа и понижаются (0.4x / 0.35x). v6 добавит автоматическую половину: **семантическое** обнаружение противоречий (без объявления), мультистраничный синтез при инжесте и автофайлинг ответов — память, которая сама разрешает противоречия и не гниёт.
